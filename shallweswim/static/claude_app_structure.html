React app architecture · /app

full app structure options

The new React app at /app currently covers the NYC conditions hero. The existing Jinja app has a lot more on top of that — a rich currents detail page with a time-warp control, a swim-direction current map, location-specific commentary, an all-locations grid, and embeddable widgets. Each location also exposes a different subset of features (some are tidal, some are rivers, some are pools). This sheet sketches three structural directions for how to fold all of that into the React app, plus the cross-cutting building blocks they share.

Cross-cutting building blocks Reusable

Before picking a page architecture, four sub-decisions show up in all of the design candidates below. They can be mixed and matched.

1. URL structure

/app/{loc} is canonical for a location. Time-shift and modal expansion should ride as query params so they're shareable and back-buttonable.

/app/nyc
/app/nyc?planner=open
/app/nyc?at=2026-05-18T15:30:00
/app/nyc?planner=open&at=2026-05-18T15:30:00
/app/locations

?at= is the canonical app-state for the shifted planning time — location-local ISO-8601 without a timezone offset. The location code supplies the timezone; the backend rejects Z, explicit offsets, DST-ambiguous local instants, and times outside the ±24h prediction window.

2. Time-warp control variants

Existing Jinja: three buttons (−1h, now, +1h). Three richer options for the React app, all backed by the same shift query param:

Buttons ◀ −1h now +1h ▶
Presets now next slack peak flood peak ebb +3h
Slider
+1h 30m · 3:54 PM

Preset chips are the most "swimmer-natural" — people plan around slack/peak, not around clock minutes. A slider is the most powerful but harder to use precisely on touch.

3. Location switching

The current Jinja site links between locations via 3-letter codes at the bottom and a /all page. Three options for React:

  • Header dropdown — small click target, always visible, fewer routes.
  • Dedicated /locations page with widget grid — matches existing /all, scales to more locations, surfaces capability differences.
  • Persistent left rail (desktop only) — fast switching for power users who watch multiple locations.

Pick both: header dropdown for casual switching, /locations page for browsing, and the bootstrap endpoint already serves the data either approach needs.

4. Capability-driven section ordering

The page is a list of sections, each gated by a capability flag from /api/app/bootstrap (or by checking whether conditions includes the relevant payload). Sections render in a stable, location-agnostic order:

hero · water movement · forecast · live view · trends ·
currents detail · transit · sources · location nav

Missing sections collapse without leaving holes. Empty-state copy (e.g. "Chicago is a lake — no tide cycle") only shows when the section's absence might confuse a user.

Hard-coded NYC content vs API config Architecture

NYC carries most of the app's deep local knowledge — direction commentary at Grimaldo's Chair, the "currents lead tide by 2 hours here" physics note, an annotated current map with arrows at named spots, the legacy harbor charts, the Aquarium-vs-pier reversal note. The current direction has been to pipe all of that through /api/app/bootstrap so it's "configurable", but for a single frontend+backend app released together this is over-engineered. There's no version-skew scenario between the React bundle and the bootstrap config in production. Hard-code rich, locality-bound content in a per-location React module; reserve the bootstrap for operational bits.

Lives in frontend/src/locations/<loc>/

Domain content that's tightly coupled to local knowledge and unlikely to change without a code review:

  • Direction commentary ("flooding flows Coney Island → Manhattan Beach")
  • Physics notes ("currents lead the tide by 2 h at Grimaldo's Chair")
  • Custom annotated maps + overlay coordinates
  • Local landmarks and direction-flip rules ("near the Aquarium…")
  • Legacy harbor chart content / copy / arrangement
  • NOAA-station averaging explanation
  • Swim-direction recommendations driven by current phase

Lives in /api/app/bootstrap

Operational flags and links that could plausibly change without touching component code:

  • Which locations exist + default location code
  • Per-location feature flags (has temp / tide / current / webcam / transit)
  • Lat/lon, timezone, swim-location landmark name
  • Webcam URLs and channel IDs
  • Transit route IDs
  • Source citation links
  • Whether the location has static current charts (capability flag)

Three structural directions

Each design lays out the same data, just folded differently. All three handle the same per-location matrix above — the question is where the time-warp, map, and currents commentary live, and whether they get their own route.

closest to existing IA

The hub links to the detail page from a single "Current details" affordance, only rendered when the location actually has a current source. Matches the Milestone 4 plan in DESIGN.md and is the lowest-risk path.

What this gets right

  • One concern per route — the hub stays scannable; the detail page can sprawl without crowding the home view.
  • Deep-linkable. /app/nyc/currents?shift=180 is one URL that fully reproduces a future-time view.
  • Detail page can be lazy-loaded; the hub bundle stays small for the 80% of visits that just want "what's the temp?".
  • Maps directly onto the existing Jinja routes — same backend assumptions, easiest Milestone 4 lift.

What it costs

  • The time-warp / planning feature is one click away; casual visitors may never find it.
  • Two pages to design, navigate, and test.
  • Per-location adaptability lives in two places (hub + detail).
  • Mobile users have to leave the hub to plan a swim time.

Design 2 · Location Dashboard with Planner Mode

recommended

One page per location, no nested route. The page is a location dashboard that has two modes driven by URL state: right-now (default) and planner (when ?at is set). Deep currents detail lives in a responsive planner overlay (full-screen on mobile, side drawer on desktop). A sticky bar at the top transforms between a now-anchor (after scroll, in right-now mode) and the page-level time control (always visible, in planner mode). Time selection is page-level state, not overlay-local — every prediction-respecting component on the page updates together. NYC's domain depth lives inside React modules (frontend/src/locations/nyc/), not in /api/app/bootstrap. Webcam and Windy load eagerly at their normal sizes — they're primary content.

Route decision

This design replaces Design 1's dedicated /app/{loc}/currents detail route. The currents depth lives in an overlay on the location dashboard, not on a separate page.

  • Canonical URL: /app/{loc} with query params. The React app isn't bound to the Jinja URL shape, so we adopt the dashboard convention directly — no /app/{loc}/currents route at all.
  • Primary planner URL: /app/nyc?planner=open&at=… — dashboard with planner open at the chosen time.
  • Known location without prediction data: the dashboard loads normally and quietly suppresses the planner trigger and overlay. A deep-linked ?planner=open URL still resolves — the param is a no-op rather than an error.
  • Unknown location code: hard HTTP 404. The 404 is reserved for locations the app doesn't know about, not for known locations that simply lack a planner.

URL state & back-button behavior

?at= is the canonical app-state parameter for the selected planning time. The wire format is an location-local ISO-8601 timestamp without a timezone offset: ?at=2026-05-18T15:30:00. The location code supplies the timezone, and the backend rejects Z, explicit offsets, DST-ambiguous local instants, and times outside the ±24h prediction window. The UI just renders "3:30 PM". The backend's existing ?shift= minutes parameter remains accepted for relative controls and legacy callers; shift-aware endpoints now accept ?at= directly. Examples in the table below use a short form for readability.

URL (short form) Sticky bar Overlay Predictable signals
/app/nyc now-anchor (after scroll) closed right-now
/app/nyc?planner=open now-anchor (after scroll) open · scrubber at "now" right-now
/app/nyc?at=…T15:30… pinned · time control closed at 3:30 PM
/app/nyc?planner=open&at=…T15:30… pinned · time control open · scrubber at 3:30 PM at 3:30 PM

History semantics: opening the planner pushes a history entry, so the device back button predictably closes the overlay. Scrubbing time inside the overlay replaces the current entry (each tick isn't an undo step). Closing the overlay replaces (URL drops ?planner=open, keeps ?at). Tapping "reset to now" replaces (URL drops ?at). Net effect of an in-app session: back button from any shifted state returns the user to where they were before they opened the planner.

Direct-entry edge case: if a user lands on the app via a deep link like /app/nyc?planner=open&at=… (shared by someone else), the browser back button follows normal browser history — it leaves the app to the previous page entirely, not into a synthetic "without planner" entry. The close × inside the overlay still replaces to /app/nyc?at=… as expected, so the user can both dismiss in-app and back-out of the app cleanly.

Three interaction patterns

Together these replace the original "accordion of collapsed sections" sketch. None of them leave a header or empty space behind when not in use. Webcam and Windy load eagerly at their normal sizes — they're primary content, not optional.

1 · Planner overlay (responsive)

Trigger: a small inline "PLAN AHEAD →" link in the bottom row of the Water Movement card on both mobile and desktop. Once the hero has scrolled past, the same trigger also appears as a "Plan" chip on the right side of the sticky now-bar so it stays reachable.

Display: full-screen on mobile (replaces the home view; close × and title bar at the top), side drawer on desktop (slides in from the right alongside the home column).

Dismissal: close ×, ESC, click-outside (desktop only), and the device back button all dismiss the overlay. The URL time persists across close — the user doesn't lose their selection. When closed, the sticky time-bar (pattern 3 below) takes over as the ongoing time control, so the user can keep scrubbing without re-opening the overlay. The overlay is only needed again to see the deep detail (map, projection, commentary).

URL is canonical: ?planner=open tracks visibility, ?at=15:30 tracks the selected time. They're independent — a shared link without planner=open opens the page at the chosen time without the overlay; with planner=open&at=15:30 it lands inside the overlay at that time. Refresh, back/forward, and link-sharing all behave the same.

2 · In-card mode toggle

The Water Movement card carries small Now / Soon chips in its corner. Tapping Soon swaps the card's contents in-place with a 2-hour outlook — no new section, no displaced layout. For NYC, "Soon" surfaces the next slack/peak time and a one-line direction recommendation ("ebb peaks at 5:42 — plan for a swim toward Coney Is").

3 · Sticky bar · now-anchor ↔ planner control

Same DOM element, two modes driven by URL state.

Normal (no ?at): after the hero scrolls past, a thin dark bar attaches to the top with water temp + tide chip + current chip + a small "PLAN" chip on the right. Always glanceable on long pages; absent at the top so the hero doesn't fight it. Tap the bar body to jump back to Water Movement; tap "PLAN" to open the planner overlay.

Planner mode (?at=15:30): the bar pins to the top regardless of scroll position — its presence signals "you're not looking at right now". It transforms into the page-level time control: time scrubber, preset chips (now / +1h / next slack / peak ebb), selected-time readout, and a "× reset to now" affordance. Dragging the slider updates every prediction-respecting component on the page in real-time — no need to re-open the overlay to change time.

What the planner time controls

The time selected in the planner (URL ?at=15:30) is page-level state, not overlay-local. Every component that can be predicted respects it; live / observation / historical components stay anchored to right-now and carry a small "live" badge so the user can tell which is which.

Respects ?at= · shows the future state

  • Water Movement drift bars (tide + current)
  • Tide event list (next / previous tides relative to ?at)
  • Currents detail inside the overlay (magnitude, phase, direction)
  • Tide × current projection plot (shift-aware endpoint)
  • NYC annotated current map arrows
  • Smart Water Movement copy ("Best for Coney Is → Manhattan Beach at 3:30 PM")
  • "In-card Soon mode" outlook also rebases off ?at when set
  • Sticky bar transforms into the time-control bar (scrubber + presets + reset) so the user can keep scrubbing time without re-opening the overlay
  • Future: water temperature prediction, when that feed is added
  • Future: wind / wave forecast at chosen time (if we ever back away from the Windy iframe)

Always right-now · ignores ?at=

  • Live webcam (YouTube / EarthCam / WetMet)
  • Windy iframe (carries its own time controls inside the embed)
  • Transit cards (NYC B / Q southbound — operational live state)
  • Temperature trend plots (historical, not predictive)
  • Observation-source current cards (Louisville — observations don't shift)
  • Sources, location nav, hero metadata

Each of these carries a small LIVE or NOW chip when the page is in shifted state so the visual contrast is explicit.

The sticky time-bar is doing double duty here: it is the shifted-state indicator (its mere presence says "you're not looking at right now"), and it's the time control. No separate "Viewing 3:30 PM · reset" chip is needed — the bar carries the selected-time readout and the "× reset to now" affordance directly.

Overlay interaction contract

Even with accessibility deferred for v1, the overlay needs a clear interaction contract to feel native on both viewports.

  • Open: tap the "PLAN AHEAD →" link on Water Movement (or the "PLAN" chip on the sticky now-bar after scroll). Pushes one history entry.
  • Close affordances: close × in the overlay header, ESC key (desktop), click-outside the drawer (desktop), back button (mobile + desktop). All four converge on the same outcome.
  • Page scroll lock under overlay: on mobile the home view is hidden anyway (full-screen); on desktop the dashboard column behind the drawer is scroll-locked while the drawer is open. The drawer itself scrolls independently, which matters for long NYC content — map, projection plot, direction commentary, legacy charts, and methodology notes can run past the viewport without the dashboard moving.
  • Scroll restoration on close: closing returns the user to the same scroll position they had when they triggered the overlay. Closing from a "PLAN" chip on the sticky bar restores the post-scroll position.
  • Shifted-time state is always visible: when ?at is set, the sticky time-bar is pinned at the top regardless of scroll. The user can't accidentally lose track of which moment they're looking at.
  • Touch targets: slider thumb, preset chips, and the × button are sized for 44×44 pt minimum so they're comfortable on mobile.
  • Slider responsiveness: drag updates predictions in real time as the user scrubs (debounced ~150 ms on the API-bound projection plot); preset chips snap.
  • Refresh / link-share: the URL is canonical state, so refresh, paste-into-new-tab, and shared links all reproduce the exact view.

What this gets right

  • Discoverability — the planner is one tap from the surface, but its trigger is small and unobtrusive when not in use.
  • One page · one URL · one mental model. Overlay state encodes in the URL so links are shareable.
  • The long-page concern is addressed by the sticky now-bar (keeps key info glanceable after scroll) and the planner overlay (deep content with zero page footprint when closed). Webcam and Windy stay eagerly loaded — they're primary content.
  • Page reflows minimally — no layout jank when expanding.
  • Location-specific depth lives where it belongs: NYC's annotated map only appears inside the planner sheet, not on the home surface; per-location React modules pick up the local commentary.
  • Mobile-native sheet idiom is familiar to iOS / Android users.

What it costs

  • More component surface to build — full-screen mobile overlay, desktop drawer, sticky now-bar, in-card mode toggle. Heavier engineering than Design 1's two simple routes.
  • URL state for the overlay needs careful design (?planner=open&at=...) so back/forward and link-sharing both work.
  • Mobile users who don't recognize the bottom-sheet idiom might miss the depth on first visit. Mitigated by the small "PLAN AHEAD →" affordance and by analytics-driven copy refinement after launch.
  • Accessibility load: focus trap inside the overlay, ESC to close, ARIA roles, keyboard-reachable sheet handle, screen-reader announcements for state changes.
  • Visual hierarchy needs care so the now content still reads as foreground even when the planner is open.

Design 3 · Right-Now / Planner Mode

most ambitious

Treats time as a first-class app state. The location page has two modes: Right Now (default — same as today) and Planner, toggled from the hero. In Planner mode, a time scrubber locks to the top of the page; every prediction-based widget — Water Movement, current map, tide × current plot, prose summary — updates as the scrubber moves. Time is encoded as an location-local timestamp in the URL (?at=2026-05-18T15:30:00), so a link reads as a moment in time for that location. Temperature, webcam, trends, transit stay in their right-now state.

  • Most of our visits are 30-second glances; building a full planner may be over-investment for the median use case.

Recommendation

Adopt Design 2 — Location Dashboard with Planner Mode — as a deliberate replacement for the dedicated currents detail route.

One page per location with two URL-driven modes (right-now by default, planner when ?at is set). The page handles depth through a responsive planner overlay (full-screen on mobile, side drawer on desktop), a sticky bar that transforms from now-anchor into the page-level time control, and an in-card Now/Soon toggle on Water Movement. Webcam and Windy stay eagerly loaded at their normal sizes — they're primary content. Nothing on the page is a "collapsed section header taking up vertical real estate" — triggers are small inline links and closed overlays leave no footprint.

Where NYC's local depth lives: the planner overlay delegates to frontend/src/locations/nyc/NycPlannerSheet.tsx (plus NycCurrentMap.tsx, NycDirectionCommentary.tsx, NycPhysicsNote.tsx, NycLegacyCharts.tsx). The home surface's Water Movement card pulls its smart phase-aware copy from the same NYC module — "Best for Coney Is → Manhattan Beach swims" when flooding, etc. None of that domain content flows through /api/app/bootstrap; the bootstrap endpoint stays focused on capability flags, station IDs, webcam URLs, transit route IDs, and citation links.

Time-warp control inside the overlay: replace the three Jinja buttons (−1h / now / +1h) with a preset chip bar plus a slider backed by an location-local URL parameter (?at=2026-05-18T15:30:00 alongside ?planner=open). The chips let casual visitors poke at "next slack" or "peak ebb" without arithmetic; the slider gives precision; the local timestamp makes links shareable. The existing shift minutes parameter is still accepted server-side for relative controls.

This is a deliberate replacement for the earlier Design 1 plan that had been informally headed into DESIGN.md as a dedicated /app/{loc}/currents route. That route adds navigation friction and hides the planner behind a click — and the planner is the most interesting thing this app does for NYC swimmers. The React app isn't bound to the Jinja URL shape so no /app/{loc}/currents route exists in React at all; the dashboard URL with query params is the only per-location surface. Design 3's full-page "Planner mode" toggle is also out — too much new chrome and state for a 30-second-glance app; the URL-state-driven mode swap that Design 2 already has does the same job with less ceremony.

Open questions worth resolving before adopting this in DESIGN.md

  • Where exactly to draw the line between locations/nyc/ hard-coding and the bootstrap schema? Some items (lat/lon, swim-location name, webcam URL) are obviously bootstrap; others (direction commentary, physics notes) are obviously a React module. Swim-location landmark names and a couple of map-overlay coordinates sit in the middle — decide once, document on the bootstrap endpoint's schema docstring, and stop relitigating.
  • Should /app/locations ship in Milestone 4 (matching /all in the Jinja site), or be deferred to a later pass with a header dropdown carrying the bootstrap location list in the meantime?
  • Embed widget (/{loc}/widget) — does it move into /app at all, or stay on the Jinja site as a "legacy embed" surface for existing iframe integrations?
  • For Louisville (river, observation currents), what does locations/sdf/SdfCurrentsDetail.tsx contain? Probably just the prose summary + tide-less observation plot with no time-warp controls — but its content is local knowledge too, so it deserves its own module rather than landing in the generic fallback.
  • When a location has no current data at all (Chicago, Austin, Boston, San Diego, Seattle, San Francisco), a deep-linked ?planner=open URL is decided as a no-op (the dashboard loads normally with the planner suppressed). Confirm that's the right call versus stripping the param on render — the latter would change the URL the user sees, which has its own confusion tradeoff.
  • Should the empty-state copy ("Chicago is a lake — no tides here") live in the generic location module, or get one location-specific override per place? Lake / river / spring / ocean each warrant slightly different phrasing.

Shipping path Milestones

Recommendation alone isn't a plan. Four frontend milestones, each independently shippable, ordered so the most-used surface ships first. Each milestone is FE-only against existing endpoints unless noted; backend evolution is a parallel track captured below.

M1 · Planner overlay (the foundation)

Build the responsive planner overlay for NYC — full-screen mobile, side drawer desktop. Inside the overlay: time scrubber + −1h / now / +1h preset chips, annotated Coney Island map, tide × current projection plot, NYC direction commentary, physics note, legacy harbor charts. "PLAN AHEAD →" trigger on Water Movement. All location-specific content hard-coded in locations/nyc/.

Canonical at ↔ existing shift translation: the React app URL uses ?at= (ISO timestamp) as canonical state. Backend endpoints accept /api/{loc}/currents?at=… and /api/{loc}/plots/current_tide?at=… directly. Existing inbound links with ?shift= can still work because shift remains supported.

Result: feature parity with the Jinja currents page, plus the dashboard / overlay model. No backend changes required.

M2 · Sticky shifted-time bar

Add the sticky bar that transforms between now-anchor mode (after scroll) and planner-mode time control (always pinned when ?at is set). Becomes the ongoing time handle so the user can keep scrubbing without re-opening the overlay. Page-wide: every prediction-respecting component on the dashboard now updates from the bar's slider, not just inside the overlay. No backend changes.

M3 · In-card Now/Soon + smarter presets

Add the Water Movement card's Now / Soon chips for the 2-hour outlook. Add "next slack" / "peak flood" / "peak ebb" preset chips to both the overlay scrubber and the sticky time-bar. Requires a backend addition: named inflection-point timestamps in /api/{loc}/currents (otherwise the FE has to compute them, which is brittle).

M4 · Generalize to other locations

Build locations/sdf/ (Louisville river — no planner trigger, "Recent flow" card instead), locations/chi/ (lake — no Water Movement card), locations/aus/ (spring — minimal), locations/generic/ (fallback for any future tidal location). Each module selects which generic components render. Also picks up the router-level work formerly tracked as a "Generalize Location Pages" milestone — /app/:locationCode routing for any registered location, per-location feature-flag rendering, graceful absence of features, and 404 / unavailable state handling.

Parallel backend track — optional, additive: (a) named inflection-point timestamps in /api/{loc}/currents (unblocks M3's smart presets); (b) replace the plots/current_tide image endpoint with a JSON time-series so the projection plot can render client-side (interactive tooltips, sharper labels); (c) accept the new ?at= param server-side alongside ?shift=. None of these block M1 or M2.

M1 alone gets NYC users 90% of the way to feature parity. M2 unlocks the planner-mode UX that the dashboard model depends on. M3 and M4 are quality-of-life — they can wait until the React app has displaced the Jinja site.
