React app migration workshop

incorporating the full HTML app

A standalone design artifact for moving the richer existing HTML experience into the React app: current detail pages, maps, plots, local swim commentary, and time shifting, while keeping the app adaptable across tidal beaches, rivers, webcam-only locations, and simpler temperature/tide locations.

North Star

Keep the home page as a quick answer. Put deeper, location-specific interpretation in capability-aware detail pages that reuse backend APIs and degrade cleanly when a location lacks predictions, maps, tides, webcams, transit, or static chart assets. For NYC, treat local swim knowledge as product content that can live directly in the React app when that is simpler than building a generalized configuration system.

The app is a single frontend/backend release, so not every piece of location-specific presentation has to flow through bootstrap. Use the API for live/predicted values and asset filenames. Put stable editorial knowledge, map labels, and swim-direction interpretation in app code when that is clearer.

Information Architecture Options

Three plausible ways to fold the old current page into the app. The recommendation is a hybrid of option A and option C, with the option B overlay pattern kept as a later enhancement if the dedicated route starts to feel too disconnected from the Today surface.

A. Dedicated Current Detail Page

Keep the main page light. Current details live at /app/:location/currents.

Entry

C. Capability-Aware Detail Hub

Detail pages share a module system: current, tide, temperature, forecast, webcam, and local notes. NYC activates more modules; river locations activate different modules.

Routes
Start with /app/:location/currents. Later add /app/:location/temperature or /app/:location/planning only if needed.
Modules
Summary, time controls, projection plot, map, local guidance, source notes, fallback explanation. For NYC, the local guidance module can be imported directly from app code instead of described by API metadata.
Decision
Use this as the implementation model even if the initial UI exposes only the currents page.

Implementation Shape

A practical path that preserves the existing backend split and avoids moving NOAA/USGS/plot logic into the browser.

Frontend modules

Capability-aware shell, with explicit NYC-specific content modules where useful.

CurrentsDetailPage Replaces the placeholder route. Reads shift from search params and calls /api/:location/currents.
Location detail registry Selects optional app-side modules by location code: nyc gets the custom guide and map overlays; other locations fall back to generic tidal or river detail components.
TimeShiftControls URL-backed controls: now, -1h, +1h, and later presets like next slack / next peak when the API exposes them.
CurrentMapPanel Renders only when current_chart_filename is present. Otherwise omitted or replaced by source context.
NycCurrentGuide A normal React component, not API data. Contains the local swim-direction explanation, Aquarium reversal warning, map overlay captions, and practical route advice.
ProjectionPlot Uses existing navigation.plot_url or builds the current-tide plot URL with shift.

API / data needs

Most of phase one can use existing endpoints.

Phase 1: no backend rewrite Use current response fields: current state, navigation, plot_url, current_chart_filename, and legacy_chart.
Phase 2: app-side NYC content Implement stable NYC local knowledge directly in React. Use small hard-coded maps/labels/caveats where that is simpler than pushing one-off content through bootstrap.
Phase 3: optional planner overlay If the route feels too separated from Today, mount the same currents detail modules inside a responsive planner sheet. Keep routing/search params as the canonical state.
Phase 4: richer prediction events Add next slack / next peak / time-to-flip fields so the UI can offer smarter time warp presets than fixed one-hour steps.
Phase 5: structured curves Replace or supplement image plots with sampled tide/current series only if interactive charting becomes worth the added API surface.

Questions To Resolve

These are the few decisions that should be explicit before the React current detail work becomes implementation work.

Unsupported routes

What should /app/:location/currents do when a location has no current detail?

My preference: no entry link from Today, and if someone lands there directly, show a concise unsupported detail state with a route back to the location page. Avoid a hard 404 unless the location itself does not exist.

River detail model

Louisville and future rivers should not inherit tidal language.

Build a separate observation-detail variant when it is needed: live velocity/flow, recent trend, source freshness, and safety context. No time warp unless there is a real forecast source.

What remains outside React

The old HTML app, embeddable widgets, and static charts can coexist during the migration.

Keep the old HTML current page alive until the React route has parity for NYC. Move only the user-facing app experience first; decide separately whether older widgets need to become React or stay as stable server-rendered surfaces.
